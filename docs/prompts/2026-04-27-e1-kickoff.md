## Tradeoffs

### `findFirst({where:{packageId},orderBy:{createdAt:'desc'}})` vs `findUnique`

- `packageId` no tiene `@unique` en `prisma/schema.prisma`. Prisma 7 rechaza `findUnique` en columnas no únicas → tipos rotos en compile time.
- Un mismo paquete puede generar múltiples `PackageEvent` en E1 (`transit`, `received`, `delivered` con el mismo `pkgId`). `findFirst + orderBy createdAt desc` devuelve el evento más reciente, lo que **ya prepara** la respuesta para RF01 E1 ("identificar la última acción realizada por la ciudad").
- Alternativa rechazada: agregar `@unique` a `packageId`. ❌ Rompe E1 cuando hayan múltiples eventos por paquete.

### No agregar índice DB en `packageId` ahora

- El índice acelera la query corregida, pero requiere migración Prisma standalone.
- Mantener atomicidad del hotfix: una sola intención por PR.
- Se hace en M2 junto con el resto de migraciones del nuevo modelo (`Route`, `AuditEvent`, índices).

### `architecture.md` como stub vs UML formal aquí

- RDOC01 (3p) exige diagrama UML de componentes con estereotipos, interfaces named, NFRs anotados (per AY3 y CLAUDE.md global). Esto requiere conocer el sistema de ruteo y la auth ya diseñados, los cuales son trabajo de M2/M3.
- Producir el UML ahora contaminaría el alcance del PR M1 con decisiones aún no tomadas (Auth0 vs Cognito, exact API Gateway shape).
- Stub satisface BMAD ahora (estilos elegidos, NFRs priorizados, Mermaid inicial); la formalización viene en M3.

### README en español, code blocks en inglés

- Alineado con CLAUDE.md global del owner (`Idioma: español en explicaciones, inglés en código y comentarios de código`).
- Usuarios del repo (compañeros del grupo) lo leerán en español; los comandos/configs no se traducen.

### No tocar el controller

- El enunciado fija la URI literal `{url}/packages/{:id}`. Renombrar el path param a `:packageId` rompería el contrato.
- Renombrar la variable interna a `packageId` sería puramente cosmético y dejaría inconsistencia con la ruta.
- El parámetro pasa transparente al service; no hay lógica que dependa del nombre.

### No actualizar el connector al broker E1 ahora

- El connector hoy escucha `observer.XX.q` con `type: package-received` (E0). E1 cambia a `city.<code>.q` con `type: package-transit` + ACK/NACK + retry Fibonacci.
- Estos cambios son ~150 LoC + tests + retry policy. Mezclarlos con el hotfix RF2 dispara el blast radius del PR.
- Se trackea como Milestone M2; la PR M2 usa el repo ya con la metodología BMAD activa (este PR la deja en pie).

### Sin commit ni branch creada por el agente

- El owner decide cuándo crear `feature/e1-kickoff` desde `develop` y abrir el PR. Esto evita que el agente IA tome decisiones de gitflow que requieren revisión humana (especialmente porque `main` debe quedar protegido por 2 reviewers según enunciado).
