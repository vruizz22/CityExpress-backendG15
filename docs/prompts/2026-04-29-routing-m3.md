### Contexto del sistema

Somos una ciudad del ecosistema CityExpress. Nuestra ciudad tiene un `cityId` (code de 3 letras, ej: `RNC`). Nos comunicamos con otras ciudades y con una `central` mediante **colas RabbitMQ** usando el binding `city.{cityId}`.

### Schema base de mensajes

Todo mensaje intercambiado tiene esta estructura mínima:

```json
{
  "idpk": "uuid",        // llave de idempotencia
  "msgId": "uuid",       // ID único del mensaje (aleatorio)
  "type": "...",         // tipo del mensaje
  "timestamp": "ISO8601",
  "cityId": "RNC"        // incluir cuando nosotros enviamos
}
```

- `idpk` garantiza idempotencia: reintentos del mismo `msgId` deben tener el mismo `idpk`. Debe diferir de `msgId`.
- **ACK**: confirmar recepción de una petición → `{ type: "ack", ... }`
- **NACK**: mensaje malformado → `{ type: "nack", ... }`. No se hacen ACK de ACK.

### Rutas y tabla de distancias

Al iniciar el servicio, se solicita **una sola vez** la tabla de distancias a la `central`:

```json
{ "type": "request", "data": { "ask": "distance-table" } }
```

La respuesta contiene un objeto `distances` keyed por `cityCode`. El único campo relevante por ahora es `enabled: boolean` — indica si existe ruta directa a esa ciudad.

La tabla se actualiza periódicamente: el servicio debe escuchar mensajes de tipo `distance-table` en su propia cola y actualizar el estado en memoria (o persistente) sin volver a pedirla.

### Lógica de tránsito de paquetes

Los paquetes llegan a nuestra cola con `type: "package-transit"`:

```json
{
  "type": "package-transit",
  "packageBody": {
    "id": "uuid",
    "deliveryStrategy": "direct",
    "maxHops": 5,
    "createdAt": "ISO8601",
    "deliverNotBefore": "ISO8601",
    "originId": "central | CITY_CODE",
    "destinationId": "CITY_CODE",
    "metaContent": "",
    "isMetaEncrypted": false,
    "constraints": {},
    "priorityClass": "medium",
    "payment": 0
  }
}
```

Al recibir un paquete, el flujo de decisión es el siguiente:

#### 1. ¿El paquete es para nuestra ciudad? (destinationId === nuestro cityId)
- Reportar `received` al auditor (central).
- Si `now >= deliverNotBefore` → entregar → reportar `delivered`.
- Si aún no es momento → guardar de forma **persistente** y entregar cuando corresponda.

#### 2. ¿No es para nosotros?
- Si `maxHops === 0` → **expirado**: reportar `expired` a central. No reenviar.
- Si `maxHops > 0` → decrementar `maxHops` en 1 y reenviar:
  - **Ruta directa disponible** (`enabled: true` en tabla): enviar directo a `city.{destinationId}` → reportar `transit` a central con `nextCityId`.
  - **Sin ruta directa**: elegir una ciudad aleatoria con `enabled: true`, **excluyendo** la ciudad origen del mensaje y la `central` → enviar ahí → reportar `transit-redirect` a central con `nextCityId`.
  - **Sin ninguna ruta disponible**: guardar el paquete de forma **persistente** hasta tener rutas disponibles.

> Siempre hacer **ACK/NACK** al remitente antes de procesar.

### Reportes al auditor (central)

Cada acción exitosa sobre un paquete debe reportarse a la central **inmediatamente después** de ejecutarse:

| Evento                      | `type`           | Campos extra              |
|-----------------------------|------------------|---------------------------|
| Reenviado con ruta directa  | `transit`        | `pkgId`, `data.nextCityId`|
| Reenviado sin ruta directa  | `transit-redirect` | `pkgId`, `data.nextCityId`|
| Expirado                    | `expired`        | `pkgId`                   |
| Recibido (aún no entregado) | `received`       | `pkgId`                   |
| Entregado al cliente        | `delivered`      | `pkgId`                   |

## Output
- Contratos `MessageBrokerService` (`send`, `subscribe`) + implementación noop para no bloquear arranque.
- `PackageService` con ACK/NACK previo, ruteo directo/redirección, expiración por `maxHops`, entrega local y cola pendiente.
- `DistanceTableService` con request inicial y actualización por mensajes `distance-table`.

- Tests unitarios para decisión de entrega, `maxHops` y selección de ruta.
- Listener para `package-transit` y `distance-table` usando `MessageBrokerService`.
- Validación con Zod para mensajes y DTOs.
- Persistencia de eventos antes de ACK, con detección de duplicados.
- Persistencia de `senderCityId` para redirecciones pendientes.


## Decisión
1. Reutilizar `PackageEvent` para pendientes (`pending-*`) evitando migraciones ahora.
2. Usar broker/delivery noop para mantener el backend ejecutable mientras RabbitMQ no está operativo.
3. Excluir el `cityId` emisor y `central` al elegir redirección aleatoria.
4. Introducir Zod para validación tipada en DTOs y mensajes de broker.
5. Persistir evento inbound en `PackageEvent` antes de ACK para cumplir disponibilidad.
6. Guardar `senderCityId` como columna opcional en `PackageEvent` para evitar loops.

## Tradeoffs
- Reutilizar `PackageEvent` actualiza el `type` del evento si el `idpk` ya existía; simplifica persistencia pero sacrifica el tipo original.
- Broker noop evita fallos de arranque, pero no valida integración real hasta conectar RabbitMQ.
- Request inicial de tabla de distancias se dispara en `OnModuleInit`; en noop sólo queda registrado como advertencia.
- Agregar Zod incrementa dependencia, pero estandariza validación requerida por arquitectura. Además. al usar pnpm no es tan costosa la decisión.
- Persistir antes de ACK agrega una escritura adicional por mensaje, a cambio de resiliencia.
- Columna `senderCityId` añade complejidad al esquema, pero evita redirecciones erróneas, además de asegurar un mejor funcionamiento del programa al redirigir.
